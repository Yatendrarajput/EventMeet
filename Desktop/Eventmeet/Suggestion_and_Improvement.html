2. Functional Requirements — Planned vs Built

FR1 — Authentication + Email Verification
FULLY MET

Master: Register → email_verified_at NULL → verify email → can browse. Cannot flag availability or book until verified. JWT 15min, refresh 30 days, bcrypt rounds 12.

✓ All 8 auth endpoints built. Email verification guard on availability + connections + bookings. JWT 15min access, 30-day refresh rotation. bcrypt rounds 12. Redis blacklist on logout.
FR2 — Event Discovery
FULLY MET

Master: city/category/date filters. Redis cache 5min TTL. Published events only. view_count incremented on detail fetch.

✓ Browse with category + city + date filters. Redis cache 5min working. Published-only filter in place. view_count incremented on GET /events/:id. tags[] filtering also added (v1.2 requirement).
FR3 — Availability
FULLY MET

Master: Flag/remove per event, note optional, expires at event start_datetime, BullMQ hourly expiry job, blocked users filtered, requires email verified.

✓ Flag + remove endpoints. Note field. BullMQ expiry job built. Block filter on available-users. Email verification required guard. Added GET /events/:id/my-availability (not in master — useful for frontend "am I going?" state).
FR4 — Connections
CHANGED — API RESTRUCTURED

Master defined 3 event-scoped endpoints: POST /events/:id/connections, GET /events/:id/connections, PATCH /connections/:id.

Master: POST /events/:eventId/connections (send), GET /events/:eventId/connections (list), PATCH /connections/:id (respond)
Built: POST /connections/events/:eventId/users/:userId (send), GET /connections (my accepted), GET /connections/requests (received), GET /connections/sent (sent), PATCH /connections/:id (respond)

Why changed: The frontend needed separate paginated endpoints for each tab (connections/requests/sent) which the master's single list endpoint didn't provide cleanly. The added endpoints serve the Connections page tabs. This is an additive change — the core send/respond flow is preserved.

FR5 — Real-time Chat
PARTIAL — BOOKING_INVITE TYPE NOT WIRED

Master: Socket.io, DIRECT + GROUP types, event-scoped, PENDING for Quick Connect, message types: TEXT / SYSTEM / BOOKING_INVITE / QUEUE_UPDATE. Max 2000 chars.

✓ Socket.io working. DIRECT + GROUP conversations. TEXT + SYSTEM messages working. 2000 char limit enforced at DB and middleware. Real-time delivery working.
✗ BOOKING_INVITE message type: planned as an interactive card in chat that triggers booking invite flow. Not connected because the full group booking flow (QUEUE_PENDING stage) was simplified. QUEUE_UPDATE messages also not live for same reason.
FR6 — Credits + Razorpay
INCOMPLETE — CORE PAYMENT NOT LIVE

Master: Recharge via Razorpay only. Internal credits. Non-withdrawable. 1-year expiry. Idempotent via razorpay_payment_id UNIQUE constraint. Credits deducted at booking.

✓ credit_balances, credit_transactions tables built. GET /credits/balance, POST /credits/recharge/initiate, POST /credits/recharge/webhook endpoints built. HMAC signature verification on webhook built. Idempotency key logic built.
✗ Razorpay account not set up — no live API keys. Webhook not receiving real events. Booking payment currently creates PAYMENT_PENDING with rupee finalAmount but does NOT deduct from credit_balances. The 9-step atomic transaction from master (SELECT FOR UPDATE on credit_balances + event_sections) is built but not triggered from the frontend booking flow.

This is the biggest gap between master plan and reality. The payment model is a credits system internally, but it's not connected end-to-end. Everything is built — it just needs the Razorpay account and a booking flow update.

FR7 — Group Booking + FR13 — Solo Booking
SIMPLIFIED STATE MACHINE

Master defined a 7-stage state machine: INITIATED → QUEUE_PENDING → SEATS_PENDING → PAYMENT_PENDING → CONFIRMED → CANCELLED → REFUNDED.

Master stages: Initiate → Invite Response (24hr) → All Accepted → Queue Order → Confirm Queue → Section + Seats → Payment Check → Confirm / Remove / Cancel
Built stages: Create booking (PAYMENT_PENDING) → Select section → Confirm payment → CONFIRMED / CANCELLED. The INITIATED, QUEUE_PENDING, SEATS_PENDING stages and the queue management endpoints exist in the backend but the frontend booking flow skips directly to PAYMENT_PENDING. The invite → accept → queue → seats pipeline is designed but not surfaced to users.

Why simplified: The full 7-stage flow requires a group-coordinated UI (all participants must respond before proceeding). For Phase 1 with no real users, the simplified solo + small group flow was prioritised to get something testable. The backend tables support the full flow — only the frontend flow is simplified.

FR9 — Tickets
PARTIAL — AWS NOT CONFIGURED

Master: QR code (HMAC hash), PDF async via BullMQ, S3 storage, CloudFront pre-signed URL (1hr), email via SES. My Bookings page.

✓ tickets table built. QR code HMAC generation logic built. BullMQ ticket generation job built. GET /tickets, /tickets/:id, /tickets/:id/download endpoints built. My Bookings page in frontend.
✗ AWS S3 bucket not configured. No real PDF being generated. CloudFront distribution not set up. SES email not configured. The download endpoint returns a placeholder — it will return a real pre-signed URL once S3 is connected.
FR10 — Refunds
BUILT (pending payment activation)

Master: Event cancel → BullMQ async → credits 2-5 min. idempotency_key UNIQUE prevents double refunds.

✓ refunds table built with UNIQUE(idempotency_key). BullMQ refund job built and queued on event cancellation. Refund logic: credits added back to credit_balances. Will work correctly once Razorpay + credits flow is live.
FR11 — Ratings
FULLY MET

Master: 1-5 stars, tags, optional review, confirmed booking required, prompt 24hrs after event end.

✓ POST /ratings (submit with UNIQUE rater+booking guard). GET /ratings/pending. average_rating denormalised update on user. Tags array. Score 1-5 CHECK constraint. BullMQ 24hr prompt job built.
FR12 — Notifications
FULLY MET + EXTRAS

Master: 13 notification types, saved to DB first, then Socket.io emit, unread count tracked.

✓ All 13 types implemented. DB-first then Socket.io pattern followed. Unread count tracked. Added: DELETE /notifications/:id and GET /notifications/unread-count (not in master — added for frontend badge and delete UX).
FR14 — Observability
NOT BUILT

Master: Pino structured logging. CloudWatch log shipping. Sentry error tracking. Log every booking state change, payment attempt, webhook received, Redis lock, BullMQ job.

✗ None of this is implemented. The app uses basic console logs. No structured JSON logging. No CloudWatch. No Sentry. This means in production, debugging any payment failure or booking issue will be very difficult — there's no audit trail.

4. Database Schema — Planned vs Built

Master planned 21 tables. All 21 tables were built and match the v1.2 schema definition. The schema is the strongest part of the implementation — the most faithful to the master document.

Events & Venues venues, events, event_sections, event_availability MATCH v1.2 schema changes applied: available_seats + total_seats removed from events, tags[] + view_count added. All partial indexes built. Seat constraint CHECK(available_seats >= 0) in place.

Credits & Payments credit_balances, credit_transactions, discount_rules MATCH All 3 tables built exactly to spec. UNIQUE(razorpay_payment_id), CHECK(balance >= 0), append-only ledger — all in place. Discount_rules table complete but discount calculation not integrated into booking flow.
Bookings & Tickets bookings, booking_participants, booking_invites, tickets, refunds MATCH All 5 tables built. payment_idempotency_key UNIQUE added as per v1.2. Full status enum (INITIATED → QUEUE_PENDING → SEATS_PENDING → PAYMENT_PENDING → CONFIRMED → CANCELLED → REFUNDED) in schema. Queue position and seat assignment columns present. Refunds table with UNIQUE(idempotency_key).

7. Suggestions & Improvements (Phase 2)

These suggestions are ordered by priority. The first 5 are blockers for going live. The rest are improvements that would take the project from "portfolio" to "production-grade."
1
Complete End-to-End Payment Flow CRITICAL — Blocker for Live
Phase 2 — Core Feature

The master document's entire payment model — credits, Razorpay, atomic deduction — is built in the backend but not connected. This is the most important thing to finish before going live.

  • Create Razorpay test account → get API keys → add to .env
  • Wire the booking payment flow to call the 9-step atomic transaction in booking.service.ts (SELECT FOR UPDATE on credit_balances + event_sections)
  • Test the Razorpay webhook with ngrok locally before deploying
  • Add credit balance display to the frontend (Profile page or dedicated Credits page)
  • Seed test users with credit balances to enable end-to-end booking tests
// The full 9-step atomic transaction already exists in booking.service.ts // Just needs to be called from the payment endpoint + frontend flow POST /bookings/:id/payment → triggers the atomic TX → CONFIRMED POST /credits/recharge/initiate → returns Razorpay order_id POST /credits/recharge/webhook → verifies HMAC → adds credits (already built)
2
Deploy to AWS (EC2 + RDS) — Make It Live CRITICAL — Blocker for Live
Phase 2 — Infrastructure

The master planned AWS EC2 + RDS + Docker from the start. The docker-compose.yml is ready. This is a configuration task, not a code task.

  • Launch EC2 t2.micro (free tier 12 months) — install Docker + Docker Compose
  • Launch RDS db.t3.micro PostgreSQL 15 — get connection string
  • Update .env for production (DATABASE_URL, REDIS_URL, JWT secrets)
  • Run docker-compose up -d on EC2 — server + Redis run as containers
  • Deploy frontend to Vercel — connect GitHub repo, one click
  • Point Vite proxy to production API URL
  • Set up Cloudflare DNS (free) for custom domain

Order: RDS → EC2 (deploy backend) → Vercel (deploy frontend) → verify health check → run smoke tests.

3
Set Up GitHub Actions CI/CD Pipeline CRITICAL — Blocker for Live
Phase 2 — DevOps

The master specified a 5-stage pipeline. This protects against broken deploys and is required before sharing the project publicly.

# .github/workflows/ci.yml — planned pipeline on: push + PR to main Stage 1 — CI: npm ci npx tsc --noEmit # TypeScript check npx jest # Run tests npm audit --audit-level high # Fail on HIGH/CRITICAL vulns npx eslint . Stage 2 — Build: docker build -t eventmeet-server . docker tag eventmeet-server [ECR-URL]:${{ github.sha }} docker push [ECR-URL]:${{ github.sha }} Stage 3 — Deploy to EC2: ssh into EC2 docker pull [ECR-URL]:${{ github.sha }} docker-compose up -d --no-deps server

Frontend (Vercel) auto-deploys on GitHub push — no extra config needed for that.

4
Write Integration Tests for Booking + Auth Flow HIGH — Before CI/CD
Phase 2 — Quality

The master required Jest tests to pass before any deploy. The booking flow has concurrency logic (distributed locks, SELECT FOR UPDATE) that is impossible to verify manually. Minimum test set:

  • Auth tests: register → verify email → login → get access token → refresh → logout
  • Booking concurrency test: 2 concurrent requests to book the last 1 seat — verify only 1 succeeds and the other gets a clean error (not a data corruption)
  • Payment idempotency test: Send same payment request twice with same idempotency key — verify credits deducted only once
  • Refund idempotency test: Trigger refund job twice for same cancelled booking — verify credits added only once
  • Notification creation test: Accept a connection → verify notification created in DB + socket event emitted

Use Supertest for HTTP-level integration tests against a test database. These 5 tests would cover the 3 most critical paths in the application.

5
Configure AWS SES + S3 for Emails and Tickets HIGH — Core UX
Phase 2 — Infrastructure

Email verification currently sends no real email — users cannot verify their accounts in production. Tickets cannot be downloaded. Both are free tier AWS services.

  • Verify a sender domain in AWS SES (sandbox → production request)
  • Add SES credentials to .env — email.service.ts is already built
  • Create S3 bucket for ticket PDFs — private, no public access
  • Set up CloudFront distribution pointing to S3 — for pre-signed URL generation
  • Wire the BullMQ ticket generation job to actually produce a PDF (use pdfkit or puppeteer) and upload to S3
6
Add Pino Logging + Sentry Error Tracking HIGH — Production Visibility
Phase 2 — Observability

Without structured logging, debugging a failed payment in production is nearly impossible. Pino is a 15-minute setup. Sentry's free tier covers 5,000 errors/month.

// Replace console.log with Pino (one-time setup) import pino from 'pino' export const logger = pino({ level: process.env.LOG_LEVEL || 'info' }) // Log every booking state transition logger.info({ bookingId, fromStatus, toStatus, userId }, 'Booking state changed') // Log every payment attempt logger.info({ bookingId, amount, userId, idempotencyKey }, 'Payment attempt') // Sentry — wrap Express app (3 lines) Sentry.init({ dsn: process.env.SENTRY_DSN }) app.use(Sentry.Handlers.requestHandler()) app.use(Sentry.Handlers.errorHandler())
7
Build the Full Group Booking State Machine in Frontend HIGH — Core Feature
Phase 2 — Core Product

The backend supports the complete 7-stage booking flow. The frontend only uses PAYMENT_PENDING → CONFIRMED. The group booking experience — which is EventMeet's core differentiator — is not accessible to users yet.

  • INITIATED stage: Show "Waiting for participants to accept" screen with 24hr countdown
  • QUEUE_PENDING stage: Show queue reorder UI — drag-and-drop position management
  • SEATS_PENDING stage: Show section selector with auto-assigned seat preview
  • PAYMENT_PENDING stage: Show per-person amount, credit balance check, "who is short" display
  • Wire BOOKING_INVITE socket event to show interactive booking invite card in chat
8
Add Admin Event + Venue Management UI HIGH — Required for Launch
Phase 2 — Admin

Currently, events and venues can only be created via direct DB access or Prisma Studio. The master planned POST /admin/events, POST /admin/venues, POST /admin/discount-rules. These are needed before launch — there's no way to onboard content otherwise.

  • Add "Create Event" form to Admin Dashboard — maps to POST /admin/events
  • Add "Create Venue" form — maps to POST /admin/venues
  • Add discount rule management — maps to POST /admin/discount-rules
  • Add PATCH /admin/events/:id for publish/cancel actions
9
Write a Prisma Seed Script MEDIUM — Developer Productivity
Phase 2 — DX

The master required a seed script runnable with npx prisma db seed that resets the database to a clean, testable state in 10 seconds. Currently, test data is added manually via Prisma Studio — which means a fresh environment takes hours to set up.

// prisma/seed.ts — should create: 10 users (mix verified + unverified, 2 admins, various cities) 3 venues (Mumbai, Delhi, Bangalore with sections) 8 events (mix PUBLISHED + DRAFT, various categories) 3 sections per event (Floor, Balcony, VIP — different prices) Connections (some ACCEPTED with chats, some PENDING) 5 conversations (mix DIRECT + GROUP) Credit balances (various amounts — some sufficient, some insufficient)
10
Create a Shared Types Package packages/types MEDIUM — Architecture
Phase 2 — Code Quality

The monorepo's biggest missed opportunity. Backend and frontend both define the same types independently — they drifted 5 times (isBanned vs isActive, initiator vs user, isAdmin missing, sendPaginated shape, fullName nullable). A shared package fixes this class of bug entirely.

// packages/types/src/index.ts export interface ApiUser { id: string email: string fullName: string | null isAdmin: boolean isActive: boolean isVerified: boolean } export interface PaginatedResponse<T> { data: T[] pagination: { page: number; limit: number; total: number; hasMore: boolean extra?: Record<string, unknown> } } // apps/server — imports the same type // apps/web — imports the same type // Change once → both sides update automatically
11
Fix Git Branch Strategy to Match Master Plan MEDIUM — Process
Phase 2 — Git Hygiene

The master defined: main / staging / dev / feature/* / hotfix/*. Currently only main and feature/frontend exist — and feature/frontend hasn't been merged yet.

  • Merge feature/frontend → main (squash or merge commit)
  • Create develop branch from main — all new feature branches cut from here
  • Create staging branch — auto-deploys to staging EC2 when CI passes
  • Protect main branch — require CI pass + PR review before merge
  • Future work: feature/payments, feature/full-booking-flow, feature/admin-events
12
Add GET /auth/me on App Load to Refresh User Store MEDIUM — Auth Correctness
Phase 2 — Auth

The Zustand auth store is populated at login time and never refreshed. If a user's isAdmin status changes (or they get banned) while they're logged in, their session reflects old data until they sign out. This is a latent bug that will become real once users start logging in regularly.

// In main.tsx or App.tsx — call on first render useEffect(() => { api.get('/users/me').then(res => { useAuthStore.getState().setUser(res.data.data) }).catch(() => {}) // silent fail — token may already be expired }, [])
13
Define Reusable Prisma Select Constants LOW — Code Quality
Phase 2 — Refactoring

Every service file defines its own inline select: { id: true, email: true, ... } objects. When a field needs to be added (like isAdmin: true), you have to hunt through every service to find every select. This caused Bug #19.

// apps/server/src/lib/selects.ts — define once export const userPublicSelect = { id: true, email: true, fullName: true, avatarUrl: true, isAdmin: true, isActive: true, isVerified: true, city: true, averageRating: true, } satisfies Prisma.UserSelect // In any service file import { userPublicSelect } from '@/lib/selects' prisma.user.findUnique({ where: { id }, select: userPublicSelect })
14
Add Rate Limiting Per-Endpoint as Planned in Master LOW — Security Hardening
Phase 2 — Security

The master specified granular rate limits. Currently a single global limiter is applied. The specific limits from the master are not enforced.
