1. Project Overview & Architecture

EventMeet is a full-stack monorepo application. Users discover events (concerts, sports, meetups), mark availability, get matched with other attendees, connect, book group tickets together, chat, and rate each other post-event.

5. All 12 Backend Phases

Phase 1
Foundation & Infrastructure
Done + Tested

Monorepo setup, Docker services, Prisma schema, Express server with all middleware, Socket.io + Redis adapter, BullMQ queues.

What was built

  • Turborepo monorepo with apps/server and apps/web workspaces
  • Docker Compose: PostgreSQL 16 + Redis 7 with named volumes
  • Prisma schema with 21 tables across 7 domains
  • Express app with Helmet, CORS, Pino logging, rate limiting, body parsing
  • Socket.io server with Redis adapter for horizontal scaling
  • BullMQ queues: tickets, emails, refunds, cache-invalidate
  • Zod-validated environment config with type-safe config object
  • Structured error handler middleware with AppError class
  • Response utilities: sendSuccess, sendCreated, sendPaginated
Phase 2
Authentication Module
Done + Tested

Full JWT-based auth with email verification, password reset, refresh token rotation, and Redis token blacklisting.

Key Implementation Details

  • Access token: 15-minute TTL signed with JWT_ACCESS_SECRET
  • Refresh token: 30-day TTL, stored in Redis with JTI tracking
  • Logout invalidates refresh token via Redis DEL (instant revocation)
  • Email verification token: HMAC SHA-256 hash stored in email_verifications table
  • Password reset token: same pattern, 1-hour expiry
  • Passwords hashed with bcrypt (10 salt rounds)
  • Rate limiting: 10 req/15min on auth routes
  • Anti-enumeration: forgot-password always returns 200
  • Dev mode: tokens logged to console (no email service needed)
Phase 3
Users Module
Done + Tested

User profile management with Cloudinary avatar upload, soft delete, and public profile view.

Key Implementation Details

  • Avatar upload: Multer (memory storage) → Cloudinary → URL stored in DB
  • Soft delete: sets deletedAt = now() and isActive = false
  • Public profile: limited fields exposed (no email, no sensitive data)
  • Own profile: all non-sensitive fields
  • Updatable fields: fullName, username, bio, dateOfBirth, gender, phoneNumber, city, state, interests, lookingFor
  • Username uniqueness enforced at DB level
Phase 4
Events Module
Done + Tested

Event listing, detail, availability marking, and social discovery of other attendees — with Redis caching.

Key Implementation Details

  • Event list: Redis cached (5-minute TTL) with city, category, date filters
  • Event detail: cached with section seat counts
  • Availability: users mark themselves as ACTIVE/EXPIRED for an event
  • Available users list: excludes the requesting user (social discovery)
  • Admin create/update/delete events with section management
  • viewCount incremented on each detail fetch
Phase 5
Connections Module
Done + Tested

Send, accept, decline, and remove connection requests between users at the same event.

Key Implementation Details

  • Connection is event-scoped: sender + receiver + eventId (unique constraint)
  • Accept triggers Socket.io connection:accepted event to sender
  • Notification created on new request and on acceptance
  • Connection list shows mutual connections with user profiles
  • Cannot send to self — guarded in service layer
Phase 6
Conversations & Messaging
Done + Tested

DM and group conversations with real-time Socket.io message delivery, pagination, and soft delete.

Key Implementation Details

  • Two types: DM (2 members) and GROUP (up to 15 members)
  • DM enforces unique pair — no duplicate DM conversations
  • Messages paginated in reverse-chronological order (cursor-based)
  • Real-time: new messages emitted to all room members via Socket.io
  • Soft delete: messages get deletedAt timestamp, content replaced with "Message deleted"
  • Last message preview stored on conversation for inbox listing
  • Members must be in conversation to send/read messages
Phase 7
Bookings & Ticketing
Done + Tested

Full booking lifecycle with Redis distributed seat locking, 15-minute soft reservation, Prisma transaction atomicity, and QR-coded ticket generation.

Key Implementation Details

  • Seat lock: Redis SET NX EX pattern — only one booking can proceed at a time per section
  • Re-check seats under lock to prevent race conditions
  • Prisma transaction: seat decrement + booking create + participants create — all atomic
  • 15-minute soft reservation: softReservationExpiresAt enforced on confirm
  • Confirm: booking → CONFIRMED, tickets generated with QR token EVENTMEET-{bookingId}-{userId}-{uuid}
  • Cancel PAYMENT_PENDING: seats released back to section
  • Cancel CONFIRMED: seats not released (event already sold), tickets CANCELLED
  • BullMQ ticket delivery job queued on confirm
  • Socket.io events: booking:confirmed and booking:cancelled
Phase 8
Payments (Razorpay)
Built — Razorpay account pending

Razorpay order creation, HMAC payment verification, webhook handling, and refund queue via BullMQ.

Key Implementation Details

  • Create Razorpay order from booking finalAmount (in paise)
  • Payment verification: HMAC SHA-256 of orderId|paymentId with Razorpay key secret
  • On verify success: booking → CONFIRMED, tickets generated, CreditTransaction recorded
  • Webhook endpoint for async payment events (requires Razorpay dashboard config)
  • Refund: queued via BullMQ refundQueue for async processing
  • Idempotency key on booking prevents duplicate orders
Razorpay account setup needed before this can be tested end-to-end. Keys go in .env: RAZORPAY_KEY_ID, RAZORPAY_KEY_SECRET, RAZORPAY_WEBHOOK_SECRET.
Phase 9
Notifications Module
Built

In-app notification system with 13 notification types, Socket.io real-time push, and read/delete management.

Notification Types

CONNECTION_REQUEST · CONNECTION_ACCEPTED · GROUP_INVITE · BOOKING_INVITE · QUEUE_UPDATED · SEAT_ASSIGNED · BOOKING_CONFIRMED · BOOKING_CANCELLED · PAYMENT_FAILED · EVENT_CANCELLED · RATING_REMINDER · CREDITS_RECHARGED · QUICK_CONNECT_REQUEST

Key Implementation Details

  • createNotification() is a shared internal function called by all other modules
  • Real-time delivery via Socket.io notification:new event to user's room
  • Unread count returned with every list response
  • Mark single or all as read, delete individual notifications
Phase 10
Ratings & Reviews
Done + Tested

Post-event peer rating system with score aggregation, tag-based feedback, and eligibility enforcement.

Key Implementation Details

  • Rating is booking-scoped: both rater and rated must share a CONFIRMED booking
  • One rating per rater per booking (unique constraint)
  • Cannot rate yourself
  • After every rating: User.averageRating and User.totalRatings auto-updated via aggregate query
  • Tags: up to 5 custom tags per rating (e.g. "friendly", "punctual")
  • Eligibility check endpoint: frontend can check before showing rate button
  • RATING_REMINDER notification sent to rated user
Phase 11
Credits System
Done + Tested

Double-entry credit ledger with balance tracking, atomic transactions, and internal API for other modules to add/deduct/refund credits.

Key Implementation Details

  • CreditBalance: one row per user, created on first use via upsert
  • Every credit change creates a CreditTransaction ledger entry with before/after balance
  • All operations atomic via Prisma $transaction
  • addCredits() — RECHARGE type (admin top-up, Razorpay recharge)
  • deductCredits() — BOOKING_DEBIT type (guards against insufficient balance)
  • refundCredits() — REFUND type (booking cancellations)
  • These are internal functions — consumed by bookings, payments, admin modules
Phase 12
Admin Panel
Done + Tested

Admin-only endpoints for platform management — user moderation, event control, credit top-up, and platform analytics.

Key Implementation Details

  • All routes: authenticate + requireAdmin middleware (double-gated)
  • assertAdmin() re-checks DB on every request (not just JWT claim)
  • Ban user: sets isActive=false (banned users cannot log in)
  • Cancel event: sets event CANCELLED + voids all PAYMENT_PENDING/CONFIRMED bookings in one transaction
  • Stats: live aggregate queries (users, events, bookings, revenue)
  • Admin credit top-up: calls addCredits() which creates full audit trail
Phase 13
React Frontend
Done + Tested

Full React 18 + TypeScript SPA with 11 pages, real-time Socket.io integration, TanStack Query server state, Zustand auth store, Framer Motion animations, and a complete dark design system.

Pages Built

  • Landing — public marketing page with hero + CTAs
  • Login / Register — React Hook Form + Zod validation + JWT auth
  • Events Discovery — search, category chips, city filter, paginated grid
  • Event Detail — sections picker, I'm Going toggle, People going with connect buttons
  • Profile — view + edit mode, avatar upload, interests/lookingFor chip editors
  • Connections — 3 tabs (Connections, Requests, Sent), URL-driven (?tab=), accept/decline/block
  • Conversations — WhatsApp-style two-panel, Socket.io real-time, typing indicators, delete messages
  • Bookings — list with status badges, countdown timer, QR ticket modal
  • BookingNew — 3-step flow (Review → Payment → Confirmed)
  • Notifications — real-time badge, mark read, delete, click-to-navigate by type
  • Admin Dashboard — stats overview + users table (ban/promote) + bookings list, URL-driven tabs

Key Frontend Patterns

  • sendPaginated contract: always res.data.data (array) + res.data.pagination
  • URL-driven tab state via useSearchParams (Connections + Admin)
  • Socket.io singleton: connectSocket(userId) on login, disconnectSocket() on logout
  • React Query cache cleared on logout (queryClient.clear()) — prevents cache poisoning security bug
  • Token expiry: Axios interceptor → POST /auth/refresh → retry — seamless for user. On double-fail: toast + redirect
  • Mobile layout: MobileHeader (logo + avatar → profile) + BottomNav (5 tabs with real unread badge)
  • Sonner toast library for all user feedback (success, error, session expired)

7. Key Business Flows

User Registration → Verified Account

POST /auth/register → hash password (bcrypt) → create user (isVerified: false) → generate verifyToken (HMAC SHA-256) → dev: log token to console | prod: send email via SES → return user + accessToken + refreshToken POST /auth/verify-email { userId, token } → validate token against DB → set isVerified: true, emailVerifiedAt: now() → delete token record

Event Discovery → Connection → Booking Flow

1. GET /events ← browse published events 2. POST /events/:id/availability ← mark "I'm going" 3. GET /events/:id/available-users ← discover other attendees 4. POST /connections/request ← send connection request 5. PATCH /connections/:id/respond ← other user accepts 6. POST /conversations ← start DM or group chat 7. POST /bookings ← book seats (Redis seat lock) → soft reservation: 15 min window 8. POST /payments/order ← create Razorpay order 9. POST /payments/verify ← verify payment signature → booking: CONFIRMED → tickets: generated with QR codes → BullMQ: ticket delivery job queued 10. GET /bookings/:id/ticket ← retrieve ticket + QR code

Seat Booking — Race Condition Prevention

Request A Request B │ │ ▼ ▼ Redis SET NX EX (lock) Redis SET NX EX (lock) │ ← OK (lock acquired) │ ← NULL (lock busy) ▼ ▼ Re-check seats in DB Return 409 SEAT_LOCK_BUSY │ ▼ Prisma $transaction: - eventSection.availableSeats -= N - booking.create(PAYMENT_PENDING) - bookingParticipant.createMany() │ ▼ Redis DEL (release lock)

12. Data Lifecycle — Frontend ↔ Backend

Every data fetch in the frontend follows a consistent lifecycle through these layers: Component → React Query → Axios → Backend API → Database → Redis Cache → Response → UI.

Full Request Lifecycle

User opens /events page │ ▼ React Component mounts useQuery({ queryKey: ['events', category, city, page] }) │ ▼ TanStack Query checks cache → staleTime: 2 minutes → If fresh data in cache → render immediately (no network call) → If stale or absent → fire queryFn() │ ▼ queryFn() calls Axios api.get('/events', { params: { page, limit, category, city } }) │ ▼ Axios Request Interceptor → reads accessToken from sessionStorage → attaches: Authorization: Bearer <token> → sends to: http://localhost:5000/v1/events │ ▼ Backend receives request → authenticate middleware: verifies JWT signature + expiry → rate limiter: checks Redis for request count → validate middleware: Zod schema check on query params │ ▼ events.service.listEvents() → build Redis cache key: events:city:category:date:page → redis.get(cacheKey) → if HIT → return JSON.parse(cached) → if MISS → Prisma query → PostgreSQL → redis.setex(cacheKey, 120s, JSON.stringify(result)) → return { events[], total, page, limit } │ ▼ sendPaginated() builds response { success: true, data: [ ...Event[] ], pagination: { page, limit, total, hasMore } } │ ▼ Axios Response Interceptor → on 200: pass through → on 401 (token expired): → POST /auth/refresh (sends httpOnly cookie) → get new accessToken → sessionStorage.setItem('access_token', newToken) → retry original request with new token → on 401 (refresh also fails): redirect to /login │ ▼ TanStack Query receives data → stores in query cache keyed by ['events', category, city, page] → sets background refetch timer → triggers React re-render │ ▼ Component renders → isLoading → Loader spinner → isError → error state with retry button → data → grid of EventCard components → each EventCard is a framer-motion animated div with stagger delay

Token Lifecycle — Access + Refresh

Login flow: POST /auth/login { email, password } → server validates credentials → generates accessToken (JWT, 15 min TTL) → generates refreshToken (JWT, 30 day TTL) → stores refreshToken hash in Redis: refresh:<userId>:<jti> → returns accessToken in response body → sets refreshToken in httpOnly cookie (Secure, SameSite=Strict) Frontend stores: sessionStorage.setItem('access_token', accessToken) ← refreshToken is in httpOnly cookie (never readable by JS) Every API request: Axios interceptor → Authorization: Bearer <accessToken> Token expiry (401 response): → Axios response interceptor catches 401 → POST /auth/refresh { } ← cookie sent automatically → server validates refreshToken from cookie → issues new accessToken → frontend retries original request seamlessly → user never sees a login screen Logout: → POST /auth/logout { refreshToken } → server deletes Redis key: refresh:<userId>:<jti> → frontend clears sessionStorage → redirect to /login

Real-time Layer — Socket.io

Connection: Frontend → socket.connect({ auth: { token: accessToken } }) Backend → authenticate socket via token → join room: user:<userId> Server pushes events to client: notification:new → new notification received message:new → new chat message in a conversation booking:confirmed → booking status updated to CONFIRMED booking:cancelled → booking cancelled connection:request → someone sent a connection request connection:accepted → your connection request was accepted Client listens: socket.on('notification:new', (data) => { queryClient.invalidateQueries(['notifications']) // show toast notification }) socket.on('message:new', (data) => { queryClient.invalidateQueries(['messages', conversationId]) })

13. Auth Flow — Login to Protected Page

User visits http://localhost:5173/events (unauthenticated) │ ▼ React Router renders <ProtectedRoute /> → reads Zustand authStore: { user, accessToken } → accessToken is null → redirect to /login │ ▼ User fills Login form → React Hook Form controls inputs → Zod schema validates: email format, password min 6 chars → on submit: useMutation → POST /auth/login │ ▼ Backend: POST /v1/auth/login → rate limited: 10 req / 15 min per IP → bcrypt.compare(password, hashedPassword) → generate accessToken + refreshToken → return 200 { tokens: { accessToken, refreshToken } } │ ▼ Frontend onSuccess callback → sessionStorage.setItem('access_token', token) → authStore.setUser(user) → authStore.setToken(accessToken) → navigate('/events') │ ▼ ProtectedRoute re-evaluates → accessToken now exists → render <AppLayout /> │ ▼ AppLayout renders → Sidebar (desktop) or BottomNav (mobile) + <Outlet /> → Events page mounts inside Outlet → useQuery fires → events fetched → grid rendered

Zustand Auth Store

interface AuthStore { user: User | null accessToken: string | null setUser: (user: User) => void setToken: (token: string) => void logout: () => void hydrate: () => void ← reads sessionStorage on app boot } On app mount (main.tsx): authStore.hydrate() → reads sessionStorage for existing token → if token exists: set in store → user stays logged in after refresh → if absent: store remains null → ProtectedRoute redirects to /login
