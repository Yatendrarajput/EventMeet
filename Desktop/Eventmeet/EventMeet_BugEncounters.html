Infrastructure & Setup
#1
Docker Desktop WSL Distro Timeout
Phase 1 — Setup Infrastructure High
Symptom

Docker Desktop hung indefinitely on startup. Attempting to run docker-compose up produced the error:

error listing WSL distros: running wslexec: An error occurred in the WSL subsystem

Docker Desktop showed a spinning loader forever. No containers could start.

Root Cause

WSL 2 (Windows Subsystem for Linux) — which Docker Desktop uses as its backend on Windows — had entered a corrupted/hung state. This commonly happens after Windows updates, sleep/hibernate cycles, or abrupt machine shutdowns. The WSL kernel was still registered as running but was unresponsive.

Fix Applied
# Step 1: Forcibly terminate all WSL instances
wsl --shutdown

# Step 2: Update WSL kernel to latest
wsl --update

# Step 3: Restart Docker Desktop as Administrator
# (Right-click → Run as Administrator)

After running these three steps in sequence, Docker Desktop started successfully within ~30 seconds.

Lesson Learned

On Windows with WSL 2, Docker Desktop hangs are almost always a WSL state issue, not a Docker issue. wsl --shutdown is the first command to run — it's safe and instantly resets all WSL instances. Always run Docker Desktop as Administrator on Windows.

Backend — Database & Schema
#3
Prisma Schema Circular Relation Validation Error
Phase 2 — Schema Database Medium
Symptom

Running prisma db push failed with a schema validation error:

Error: Ambiguous relation detected. The fields `sender` and `receiver`
in model `ConnectionRequest` both point to `User`.
Please provide a `name` argument in the @relation attribute.
Root Cause

The ConnectionRequest model had two relations pointing to the same User model — sender and receiver. Prisma cannot resolve which relation is which without explicit names because both back-references on User would be ambiguous.

// Before — ambiguous, Prisma rejects this
model ConnectionRequest {
  sender   User @relation(fields: [senderId], references: [id])
  receiver User @relation(fields: [receiverId], references: [id])
}
Fix Applied

Added named relations using the name argument in both the foreign model and the back-reference on User:

// After — named relations, Prisma is satisfied
model ConnectionRequest {
  sender   User @relation("connectionSender",   fields: [senderId],   references: [id])
  receiver User @relation("connectionReceiver", fields: [receiverId], references: [id])
}

model User {
  sentRequests     ConnectionRequest[] @relation("connectionSender")
  receivedRequests ConnectionRequest[] @relation("connectionReceiver")
}
Lesson Learned

Any time two fields in the same model reference the same target model, Prisma requires named relations. This pattern appears in self-referencing models (e.g. User follows User) and any model with multiple FK relationships to the same target.

#4
Windows EPERM Rename Error After prisma db push
Phase 2 — Schema Database Low
Symptom

After a successful prisma db push, the terminal printed:

Error: EPERM: operation not permitted, rename
'...query_engine-windows.dll.node.tmp' → '...query_engine-windows.dll.node'
Root Cause

The Prisma CLI tries to update its native query engine binary (.dll.node) after a schema push. On Windows, this file was still loaded (locked) by the running server process, so the OS refused the rename operation. This is a Windows-only file locking issue.

Fix Applied

No fix needed — this is a harmless warning. The schema was already pushed successfully. The existing DLL in memory is valid for the current session. The error only means the newer binary copy failed to replace the locked file. Restarting the server after the push loads the updated binary cleanly.

Lesson Learned

On Windows, always stop the dev server before running prisma db push or prisma generate to avoid file locking conflicts. On Mac/Linux this is not an issue.

Backend — Business Logic
#5
ParticipantStatus Enum Mismatch in Bookings
Phase 7 — Bookings Backend Logic High
Symptom

Creating a booking via POST /bookings threw a Prisma validation error at runtime:

PrismaClientValidationError: Invalid value for argument `status`:
Value 'CONFIRMED' does not exist in enum 'ParticipantStatus'.
Root Cause

In the bookings service, when creating the booking initiator's participant record, the code used the status value 'CONFIRMED'. However, the Prisma schema defined the ParticipantStatus enum without a CONFIRMED value. The valid values were: INVITED, ACCEPTED, DECLINED, LEFT, PAID, REFUNDED.

// Incorrect — 'CONFIRMED' does not exist in the enum
status: 'CONFIRMED'  // ← bug here
Fix Applied
- status: 'CONFIRMED', // initiator + status: 'ACCEPTED', // initiator confirms themselves status: 'INVITED', // other participants

The booking initiator is set to ACCEPTED (they've confirmed their own spot) while other invited participants are set to INVITED.

Lesson Learned

Enum values must match exactly what is declared in the Prisma schema. TypeScript catches these at compile time if you import the enum from @prisma/client rather than using raw string literals. Always use import { ParticipantStatus } from '@prisma/client' and reference ParticipantStatus.ACCEPTED instead of 'ACCEPTED'.

#6
UUID with Quotes Causing Prisma Parse Error
Phase 7 — Bookings Backend Logic Medium
Symptom

A Postman request to POST /bookings failed with:

invalid character: expected an optional prefix of `urn:uuid` followed by `[0-9a-fA-F-]{36}`
Root Cause

The UUID was copied from a previous API response and pasted into the Postman request body with surrounding double-quotes included as part of the value. So instead of a clean UUID string, the value was "\"9f32b511-825a-4341-a3e7-...\"" — Prisma's UUID parser rejected the leading quote character.

Fix Applied

Copied the raw UUID value without the surrounding quotation marks. Human error, not a code bug.

- "sectionId": "\"9f32b511-825a-4341-a3e7-abc123\"" + "sectionId": "9f32b511-825a-4341-a3e7-abc123"
Lesson Learned

When copying UUIDs from JSON responses in Postman, copy only the value inside the outer quotes, not the quotes themselves. This is a common mistake when manually assembling API test requests.

Frontend — Data Fetching (Recurring Pattern)
#7
Events Page Blank — sendPaginated Response Shape Mismatch (First Instance)
Phase 13 — Frontend Frontend / API Contract High
Symptom

The Events page rendered completely blank. Browser console showed:

TypeError: Cannot read properties of undefined (reading 'filter')

The page loaded but no event cards appeared, and filtering/searching had no effect.

Root Cause

The backend sendPaginated utility wraps array data as:

{
  "success": true,
  "data": [ ...events ],          // <-- array directly here
  "pagination": { "page": 1, "limit": 10, "total": 25, "hasMore": true }
}

But the frontend query function expected the old shape res.data.data.events — i.e., it looked for a nested events key inside data. That path was undefined, causing .filter() to throw.

Fix Applied
- return { data: res.data.data.events, pagination: res.data.pagination } + return { data: res.data.data, pagination: res.data.pagination }
Lesson Learned

This was the first of four identical bugs caused by the same incorrect assumption about the response shape. The fix pattern is always: res.data.data is the payload array, res.data.pagination is the pagination object. Establish this contract in a shared API type or document it to prevent recurrence.

#8
Connections Page Empty Despite DB Having Accepted Records
Phase 13 — Frontend Frontend / API Contract High
Symptom

The Connections page showed "No connections yet" even though the database had 2 ACCEPTED connection records for the logged-in user. Confirmed via direct DB inspection (Prisma Studio).

Root Cause

Identical to Bug #7. The connections controller also uses sendPaginated, but the frontend query assumed the response shape was { connections: [...] } nested inside res.data.data:

// res.data.data is actually Connection[] directly
// so .connections is undefined → [] → empty UI
Fix Applied

Same fix applied to the pending requests and sent requests queries in the same page.

Lesson Learned

Second instance of the same sendPaginated shape bug. At this point the pattern was identified: every paginated endpoint in the frontend needs to read res.data.data (array) and res.data.pagination (object), never treat res.data.data as an object with a named array inside it.

Frontend — Null Safety & Runtime Crashes
#11
EventDetail Page Crash — Cannot Read charAt of Null
Phase 13 — Frontend Frontend / Null Safety High
Symptom

The EventDetail page crashed with a white screen on events where some users had marked availability. Browser console showed:

TypeError: Cannot read properties of null (reading 'charAt')
Root Cause

The "People going" panel renders avatars using the first character of user.fullName as a fallback. However, fullName is a nullable field in the database — some users registered without providing a name. The code called .charAt(0) directly on a null value.

// Before — crashes when fullName is null
user.fullName.charAt(0).toUpperCase()
Fix Applied

Applied optional chaining and a nullish coalescing fallback across all avatar initial renders:

- user.fullName.charAt(0).toUpperCase() + user.fullName?.charAt(0)?.toUpperCase() ?? '?'

The same fix was applied to the organizer card and sidebar user display.

Lesson Learned

Any user-provided string field (fullName, bio, username) should be treated as potentially null/undefined in the UI, regardless of backend validation, because: (a) old records may pre-date validation rules, (b) OAuth providers may not supply all fields. Always use ?. and ?? for display values derived from user data.

Frontend — Security & State Management
#12
React Query Cache Poisoning on User Switch (Security Bug)
Phase 13 — Frontend Security / State High
Symptom

When User A logged out and User B logged in on the same browser session, navigating to /profile showed User A's personal profile data until a manual page refresh. This is a data privacy/security breach.

Root Cause

The React Query QueryClient is a singleton created once in main.tsx and lives for the entire browser session. Its in-memory cache persists across logins. When User A logs out, the cache still holds ['profile-me'], ['connections'], ['conversations'], etc. When User B logs in and visits the profile page, React Query serves the stale cached data — User A's data — before the network request for User B completes (or even fires, since staleTime was set to 5 minutes).

// main.tsx — QueryClient lives for the entire app lifetime
const queryClient = new QueryClient({
  defaultOptions: { queries: { staleTime: 1000 * 60 * 5 } }
})

// Logout — only cleared Zustand store, NOT the query cache
logout() // ← User A's data still in queryClient cache!
navigate('/login')
Fix Applied

Called queryClient.clear() before clearing the auth store and navigating away. This nukes the entire in-memory cache instantly.

+ const queryClient = useQueryClient() const handleLogout = () => { + queryClient.clear() // wipe all cached data first logout() navigate('/login') }

Why clear() over invalidateQueries()? — invalidateQueries marks data as stale but leaves it in the cache, so the stale value is still shown as a background value until the refetch resolves. clear() removes everything immediately — zero window where wrong user's data is visible.

Lesson Learned

In any SPA with a shared query cache, logout must clear all cached data. This is a commonly missed security step. React Query's queryClient.clear() on logout is the correct pattern. The same applies to any global state (Zustand, Redux) that holds user-specific data — reset all stores on logout, not just the auth token.

Frontend — Navigation & Feature Gaps
#13
Message Button on Connections Navigated Without Creating a Conversation
Phase 13 — Frontend Frontend / UX Flow Medium
Symptom

Clicking the chat icon on a connection card navigated to /conversations, but no conversation was open or pre-selected. The thread panel showed "Select a conversation." There was no way to actually start chatting with a connected user from the Connections page.

Root Cause

The message button only called navigate('/conversations') — it navigated to the page but did nothing to create or identify the conversation for that specific user pair. The conversations page had no context about which user to open a chat with.

// Before — just navigates, no DM creation
onClick={() => navigate('/conversations')}
Fix Applied

The message button now:

  • POSTs to POST /conversations with { type: 'DIRECT', eventId, memberIds: [otherId] }
  • The backend is idempotent — returns existing DM if one already exists, or creates a new one
  • Invalidates the ['conversations'] cache so the list is fresh
  • Navigates to /conversations with the conversation ID in React Router state
  • The Conversations page reads location.state.conversationId to pre-select the thread
async function handleMessage(conn) {
  const res = await api.post('/conversations', {
    type: 'DIRECT', eventId: conn.event.id, memberIds: [conn.connectedUser.id]
  })
  queryClient.invalidateQueries({ queryKey: ['conversations'] })
  navigate('/conversations', { state: { conversationId: res.data.data.id } })
}
Lesson Learned

Navigation between features needs to carry context. "Go to messages" is not enough — it must answer "which message?" When navigating with intent (open a specific item), use React Router state to pass the ID. The receiving page reads it on mount to initialize the correct view.

#14
Race Condition — Navigated Before Conversations Cache Refreshed
Phase 13 — Frontend Frontend / Async Timing Medium
Symptom

After fixing Bug #13, clicking the message button correctly called POST and navigated with the conversation ID in state. But on the conversations page, the thread still didn't open — the thread panel showed "Select a conversation" because conversations.find(c => c.id === selectedId) returned undefined.

Root Cause

A timing/ordering issue. The conversations list query used staleTime: 0 but the cache wasn't invalidated before navigating. The navigation happened while the query was still in-flight — so on mount, conversations was an empty array and selected was null. Even though the query eventually resolved, the initial render had no conversation to select.

Fix Applied

Called queryClient.invalidateQueries({ queryKey: ['conversations'] }) before navigating. This ensures the conversations page re-fetches immediately on mount and includes the newly created conversation in the list, so selected resolves on the first meaningful render.

+ queryClient.invalidateQueries({ queryKey: ['conversations'] }) navigate('/conversations', { state: { conversationId } })
Lesson Learned

When navigating to a page that depends on freshly mutated data, invalidate the relevant queries before navigating, not after. This pre-emptively marks the data as stale so the destination page fetches fresh data on mount rather than serving cached (outdated) data that may not include the new record.

#15
React Router v7 Future Flag Deprecation Warnings
Phase 13 — Frontend Frontend / Dependencies Low
Symptom

Browser console printed two yellow warnings on every page load:

⚠️ React Router Future Flag Warning: React Router will begin wrapping state
updates in React.startTransition in v7. You can use the v7_startTransition
future flag to opt-in early.

⚠️ React Router Future Flag Warning: Relative route resolution within Splat
routes is changing in v7. You can use v7_relativeSplatPath to opt-in early.
Root Cause

These are not errors — they are deprecation warnings from React Router v6 informing developers of upcoming breaking changes in v7. The app uses React Router v6 without opting into the new v7 behaviors via future flags. The warnings appear because React Router v6 ships with opt-in previews of v7's new routing behaviors.

Fix Applied

No fix required. These are informational warnings, not bugs. The application functions correctly. They can be silenced by adding future flags to BrowserRouter:

// Optional — silences warnings, previews v7 behavior
<BrowserRouter future={{ v7_startTransition: true, v7_relativeSplatPath: true }}>
Lesson Learned

Distinguish between console errors (broken functionality) and warnings (deprecation notices). React Router future flag warnings are safe to ignore until upgrading to v7. Don't spend debugging time on warnings that don't affect application behavior.

#16
"I'm Going" Toggle State Resets After Save
Phase 13 — Frontend Frontend / React State Medium
Symptom

On the EventDetail page, after toggling "I'm Going" and saving a note, the toggle appeared to reset — the button showed the wrong state on re-render despite the backend correctly saving the value.

Root Cause

The isGoing local state was derived from the initial query result but was not being synchronized after the availability mutation succeeded. The mutation invalidated the query, but the new query result wasn't driving the UI — the old local state was.

Fix Applied

Changed the component to derive isGoing directly from the query result (myAvailability?.isGoing ?? false) rather than a separate useState. After mutation, the query refetch naturally updates the UI:

// BEFORE — local state that can drift from server
const [isGoing, setIsGoing] = useState(false)

// AFTER — always reflects server state
const isGoing = myAvailability?.isGoing ?? false
Lesson Learned

Never duplicate server state in local React state. If data comes from a useQuery, derive UI state from the query result — don't copy it into useState. The single source of truth is the query cache.

#17
People Going Panel — Null fullName Crash
Phase 13 — Frontend Frontend / TypeScript High
Symptom

The "People Going" sidebar on EventDetail crashed with Cannot read properties of null (reading 'charAt') for users who hadn't completed their profile. TypeScript also started producing cascading errors: .length and .slice not found on type.

Root Cause

Two issues stacked:

  1. The AvailableUser TypeScript interface declared fullName: string but the database field is nullable — some users register without ever filling in their name.
  2. The type mismatch caused useQuery<AvailableUser[]> to fail its overload resolution, making availableUsers an unresolved type — which then broke downstream .length and .slice calls as TypeScript couldn't infer the array type.
Fix Applied

Two changes: (1) Fix the interface to match reality. (2) Use optional chaining for all display code:

// Interface fix
interface AvailableUser {
  fullName: string | null  // was: string
  ...
}

// Display fix
{user.fullName?.charAt(0)?.toUpperCase() ?? '?'}
Lesson Learned

User-provided fields (fullName, bio, username, avatarUrl, city) should always be typed as string | null in frontend interfaces — they match Prisma's nullable columns. A single wrong type can break TypeScript inference for an entire component via overload cascade.

#18

After adding URL-driven tab state to the Connections page, the entire page crashed with ReferenceError: useState is not defined at the ConnectionsTab sub-component on line ~200.

#19
isAdmin Never Returned from Login — Admin Link Never Visible
Phase 13 — Frontend Backend / Auth / Prisma Select High
Symptom

The Admin link in the sidebar never appeared — even after setting is_admin = true in the database via Prisma Studio and signing in again. The admin dashboard was unreachable from the UI.

Root Cause

Three separate omissions stacked:

  1. The Prisma select in auth.service.ts login() did not include isAdmin: true — so the field was never fetched from the DB.
  2. The AuthUser interface in auth.types.ts did not declare isAdmin: boolean.
  3. The Zustand auth store therefore always had user.isAdmin === undefined, which is falsy — the sidebar condition {user?.isAdmin && <AdminLink />} never rendered.
Fix Applied
// auth.service.ts — add isAdmin to select
select: {
  id: true, email: true, fullName: true,
  passwordHash: true, isVerified: true,
  isAdmin: true,
  emailVerifiedAt: true, isActive: true, deletedAt: true,
}

// auth.types.ts — add to interface
interface AuthUser {
  ...
  isAdmin: boolean
}

Also required: sign out and sign back in, since the Zustand store holds login-time data and does not auto-refresh from the DB.

Lesson Learned

When a field appears in the DB but not in the UI, check the chain: DB column → Prisma select → response DTO → frontend type → component condition. A missing link at any step silently returns undefined. Restarting the session is required after fixing backend auth fields — the store holds a snapshot from login time.

#20
Admin Users Tab Blank — isBanned vs isActive Mismatch
Phase 13 — Frontend Frontend / API Contract High
Symptom

The Admin Users tab loaded but showed a blank table. No console errors. Data was being fetched (Network tab showed 200 with valid user array) but the table rows were not rendering.

Root Cause

The AdminUser TypeScript interface was written with isBanned: boolean, but the backend returns isActive: boolean (active = true means NOT banned). The ban button condition u.isBanned was always undefined, causing a crash in the conditional class logic that silently swallowed row rendering.

Fix Applied
// Interface fix
interface AdminUser {
  ...
  isBanned: boolean  // WRONG — field doesn't exist
  isActive: boolean  // CORRECT — true = active, false = banned
}

// Logic fix — isActive is the inverse of "banned"
ban: u.isActive  // ban the user when they are currently active
{!u.isActive ? 'Banned' : 'Active'}  // display logic
Lesson Learned

Always verify field names against the actual API response, not intuition. "isBanned" feels natural but the backend uses the positive form "isActive". Field naming conventions (positive vs negative booleans) must be consistent and documented.

#21
Admin Bookings Tab Blank — initiator vs user Relation Name
Phase 13 — Frontend Frontend / API Contract High
Symptom

Admin Bookings tab showed blank rows. Network request returned data but b.user.fullName was undefined — rendering crashed silently.

Root Cause

In the Prisma schema, the booking → user relation is named initiator (reflecting that a booking is always initiated by one user). The admin bookings endpoint returns { initiator: { fullName, email }, event: { title }, ... }. The frontend was accessing b.user.email — a field that does not exist on the response object.

// Frontend type fix
bookings: Array<{

  user: { fullName: string | null; email: string }  // WRONG

{b.user.fullName ?? '—'}
{b.initiator.fullName ?? '—'}

Relation names in Prisma can differ from what seems obvious. "user" feels natural but the schema used "initiator" to express the semantic of the relationship. Always refer to the Prisma schema or a prior API response sample when writing frontend types for relational data.

#22
Admin + Connections URL Same for All Tabs (useState Not URL-Driven)
Phase 13 — Frontend Frontend / Routing Low
Symptom

On both the Connections page and the Admin Dashboard, switching between tabs did not update the URL. All three tabs had the same URL (/connections or /admin). Refreshing the page always reset to the first tab. Notifications couldn't deep-link to the Requests tab via /connections?tab=requests.

Root Cause

Tab state was stored in useState — local React state that lives only in memory. It does not survive page refresh and cannot be set via URL. Notification click tried to navigate to /connections?tab=requests but the page just mounted with useState's default value, ignoring the query param.

Fix Applied

Replaced useState with useSearchParams from React Router v6 for tab state on both pages:

// BEFORE — ephemeral, URL-blind
const [tab, setTab] = useState<Tab>('connections')

// AFTER — URL-driven, deep-linkable, refresh-safe
const [searchParams, setSearchParams] = useSearchParams()
const tab: Tab = (searchParams.get('tab') as Tab) || 'connections'
const setTab = (t: Tab) => {
  if (t === 'connections') setSearchParams({}, { replace: true })
  else setSearchParams({ tab: t }, { replace: true })
}

URLs: /connections, /connections?tab=requests, /connections?tab=sent, /admin, /admin?tab=users, /admin?tab=bookings

Lesson Learned

Any UI state that should survive a page refresh or be linkable from outside must live in the URL, not in component state. Use useSearchParams for tab/filter/sort state. Use useState only for ephemeral UI state (modal open/close, hover, form input).

#23
Notifications Page Blank — sendPaginated extra Field Misread
Phase 13 — Frontend Frontend / API Contract High
Symptom

The Notifications page showed a blank list even though the database contained 2 notifications and the Network tab confirmed a 200 response with data. The unread count badge also showed 0.

Root Cause

Two misreads of the sendPaginated response shape:

  1. The notification array was read as res.data.notifications — but sendPaginated always puts the array at res.data.data.
  2. The unread count was read as res.data.unreadCount — but the notifications endpoint passes it as an extra field via sendPaginated, which buries it at res.data.pagination.extra.unreadCount.
Fix Applied
// BEFORE — wrong paths
notifications: res.data.notifications,
unreadCount: res.data.unreadCount,

// AFTER — correct sendPaginated shape
notifications: res.data.data as NotificationItem[],
unreadCount: (res.data.pagination?.extra?.unreadCount as number) ?? 0,

The sendPaginated response contract (must memorize):

res.data.data             // T[] — the array payload
res.data.pagination       // { page, limit, total, hasMore }
res.data.pagination.extra // optional — endpoint-specific extra data
Lesson Learned

sendPaginated with an extra parameter places that data inside pagination.extra, not at the response root. This is a fourth variant of the sendPaginated shape confusion (after Bugs #7, #8, #9, #10) — the pattern is clear: always use res.data.data for the array and res.data.pagination for metadata. Never assume a top-level field exists outside these two.

res.data.pagination.extra  // → optional endpoint-specific data (e.g. unreadCount)
Pattern 2 — Null User Fields (Bugs #11, #17)

User-provided fields (fullName, bio, username, avatarUrl, city) are all nullable. Prevention: Always type them as string | null in TypeScript interfaces and use optional chaining in display code.

user.fullName?.charAt(0)?.toUpperCase() ?? '?'
user.avatarUrl ?? defaultAvatar
Pattern 3 — Cache Not Cleared on Logout (Bug #12)

Auth state and query cache have separate lifecycles. Prevention: Logout must always clear the query cache before clearing auth tokens.

// Correct logout order — always in this sequence
queryClient.clear()   // 1. Wipe cached data
logout()              // 2. Clear auth store
navigate('/login')    // 3. Redirect
Pattern 4 — URL-Driven vs useState Tab State (Bug #22)

Tab state that should be deep-linkable or survive refresh must live in the URL, not in component state. Prevention: Use useSearchParams for any tab/filter/sort state that has external dependencies (e.g. notification clicks that deep-link).

// Rule: if something outside the component needs to set the tab, use URL state
const [tab, setTab] = useState('overview')  // ✗ — not linkable from notifications
const tab = searchParams.get('tab') || 'overview'  // ✓ — /admin?tab=users works
Pattern 5 — Missing Fields in Prisma Select (Bugs #19)

Fields not explicitly included in Prisma select are silently omitted from the response — no error, just undefined. Prevention: When a field appears in the DB but not in the UI, check the full chain: DB → select → DTO type → frontend interface → component condition.

// Never assume a field is returned — it must be in the select
select: {
  id: true, email: true, fullName: true,
  isAdmin: true,  // must be explicit or it won't appear in the response
}
