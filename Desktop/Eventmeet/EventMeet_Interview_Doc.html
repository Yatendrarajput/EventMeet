1. Project Overview — What & Why

Before going into technical details, always start with the "what and why" — interviewers want to understand your thought process, not just what you built.

Tell me about your project. What is EventMeet?

EventMeet is a full-stack social discovery platform built around events. The core idea is: people go to concerts, sports matches, tech meetups — but they go alone or with the same 2-3 friends. EventMeet solves the social discovery problem by letting users mark themselves as "going" to an event, discover others attending the same event, connect with them, chat, and book group tickets together.

Think of it as the intersection of Eventbrite (event discovery) and Bumble BFF (social matching) — but tightly scoped around shared event attendance.

Target audience: 22–28 year olds who are active socially but want to expand their social circle through shared experiences rather than cold introductions.

What problem does this solve that existing platforms don't?

Existing platforms like Eventbrite or BookMyShow focus on ticket sales — they don't facilitate social connections between attendees. LinkedIn events are professional-only. Facebook Events lost relevance with the 22–28 demographic. There's no platform that specifically says "here are 50 people going to this Coldplay concert — connect with them before the event, book together, and meet in person."

EventMeet fills that gap. The availability feature is the key differentiator — before you even buy a ticket, you can mark interest and see who else is going, making group formation possible before the financial commitment.

2. Architecture Decisions

Why did you choose a monorepo structure?

We used a Turborepo-based monorepo with apps/server and apps/web as separate workspaces. The key benefits:

  • Shared types: TypeScript types defined once, used in both frontend and backend — no drift between what the API returns and what the frontend expects.
  • Single git history: A feature branch covers both frontend and backend changes atomically — you don't need to coordinate across repos.
  • Unified tooling: One package.json at the root, one ESLint config, one TypeScript config — less maintenance overhead.
Trade-off

Monorepos can become unwieldy at scale (hundreds of packages). For a project this size, it's the right call. At scale (Netflix, Google scale), you'd still use monorepos but with Bazel or more sophisticated build orchestration.

Why Node.js + Express over alternatives like NestJS, Fastify, or Go?

Why not NestJS: NestJS adds a lot of structure (decorators, DI container, modules) which is excellent for large teams with strict conventions. For a project where one developer is making all the decisions, it adds ceremony without proportional benefit. We replicated the module structure manually — auth.service.ts, auth.controller.ts, auth.routes.ts — giving us the same separation of concerns with less magic.

Why not Fastify: Fastify is faster than Express in benchmarks (~2x throughput) but the ecosystem maturity and middleware availability still favors Express for a full-featured application. The performance difference matters at 10,000+ req/sec — not relevant at development stage.

Why not Go: The frontend is TypeScript. Using TypeScript on the backend allows sharing of Zod schemas, types, and utilities. A Go backend would require maintaining two separate type systems.

Decision

Express + TypeScript is the pragmatic choice: massive ecosystem, team familiarity, type safety, and the ability to share code with the frontend. The architecture is modular enough to migrate individual modules to a faster runtime (Bun, Fastify) later if needed.

Explain your module structure. How is the code organized?

Each feature is a self-contained module with four files:

  • types.ts — TypeScript interfaces for inputs and outputs
  • schema.ts — Zod validation schemas for request bodies
  • service.ts — Business logic, database queries, cache operations
  • controller.ts — HTTP layer: parse request → call service → send response
  • routes.ts — Express router: URL mapping + middleware chain

This separation means the service layer has zero knowledge of HTTP — it just takes typed inputs and returns typed outputs. This makes testing easier (you can unit test the service without spinning up Express) and makes the business logic readable without parsing through HTTP boilerplate.

Why this matters in interviews

This is the repository pattern + service layer pattern. When asked "how would you add unit tests?", the answer is: mock the Prisma client, call service functions directly — no HTTP setup needed.

3. Database Design & Trade-offs

Why PostgreSQL over MongoDB for this project?

EventMeet has inherently relational data. Consider the booking flow:

  • A User initiates a Booking
  • A Booking is for an Event
  • An Event has EventSections with limited seats
  • A Booking has multiple BookingParticipants
  • Confirmed Bookings generate Tickets
  • Cancelled Bookings create Refunds

These relationships have strict referential integrity requirements — a ticket should NEVER exist without a valid booking, a booking should NEVER reference a non-existent event. PostgreSQL foreign keys enforce this at the database level, not just the application level.

MongoDB would work for the social/discovery parts (availability, notifications) where the data is document-like. But embedding booking data in MongoDB documents would either create massive documents or require manual joins in application code — which PostgreSQL does natively with JOIN queries.

Trade-off we accepted

PostgreSQL requires schema migrations — adding a new column requires a migration file and a deploy step. MongoDB is schema-less which allows faster iteration. We accepted this trade-off because the domain (payments, tickets, seats) requires the data integrity guarantees that only a relational database provides.

Why Prisma ORM over raw SQL or other ORMs like Sequelize or TypeORM?

Prisma vs raw SQL: Raw SQL gives maximum performance and flexibility. But for a TypeScript application, raw SQL results are untyped — you lose all compile-time safety. Prisma generates a fully typed client from the schema, so prisma.user.findUnique() returns a typed User object — TypeScript catches errors before runtime.

Prisma vs Sequelize: Sequelize is model-class based (define your models as classes). Prisma uses a schema-first approach — one schema.prisma file is the single source of truth for your database structure. Prisma's developer experience (auto-complete, type inference) is significantly better than Sequelize's.

Prisma vs TypeORM: TypeORM uses decorators to define schemas inline with class definitions — the schema and the TypeScript class are tangled together. Prisma keeps them separate. Prisma also has better support for complex queries and its migration system is more reliable.

Key advantage we used

Prisma's $transaction() was critical for the booking flow — we used it to atomically decrement seat counts AND create the booking record in a single database transaction. If either step fails, both roll back. This is impossible to achieve safely without transaction support.

Walk me through your database schema design. How did you handle complex relationships?

The schema has 21 tables organized into 7 domains:

  1. Identity & Trust: User, CreditBalance — core user profile and credit ledger
  2. Events: Event, EventSection, EventAvailability, Venue — event catalog with section-based seating
  3. Social Graph: ConnectionRequest — bidirectional connection with event context (connections are always event-scoped)
  4. Conversations: Conversation, ConversationMember, Message, GroupInvite — group and DM messaging
  5. Bookings: Booking, BookingParticipant, BookingInvite, Ticket, Refund — the full ticketing lifecycle
  6. Credits: CreditTransaction, DiscountRule — double-entry credit ledger
  7. Feedback: Rating, Notification — post-event ratings and notification feed

Interesting design decisions:

  • Connections are scoped to an event (eventId is required on ConnectionRequest) — this prevents random connection requests and gives context to every relationship.
  • EventSection has both totalSeats and availableSeats — we track both so we can calculate fill percentage and never allow overselling without querying all tickets.
  • Soft deletes on User and Event (deletedAt column) — data is never actually deleted, which is critical for audit trails and support tickets.
  • All PKs use PostgreSQL's gen_random_uuid() — UUIDs are generated at the database level, not application level, preventing duplicate key errors in distributed scenarios.

What are soft deletes and why did you use them?

Soft delete means instead of physically DELETE-ing a row, you set a deletedAt timestamp. The row still exists in the database but all queries filter it out with WHERE deleted_at IS NULL.

Why we used it:

  • Audit trail: If a user disputes a charge after deleting their account, their data still exists for support investigation.
  • Foreign key integrity: If a user is hard-deleted, all their bookings, tickets, ratings, messages would need to CASCADE delete or SET NULL — creating complex dependency chains. Soft delete avoids this entirely.
  • Recovery: Users who "delete" their account can reactivate it later.
  • GDPR compliance path: You can implement a scheduled job to hard-delete records older than 30 days after soft delete — giving you a grace period while still being able to purge on demand.
Trade-off

Every query must include deletedAt: null in the where clause — forgetting this is a bug that exposes deleted data. In production, you'd enforce this at the ORM middleware level (Prisma middleware) so it's applied automatically. We handle it manually per query currently.

4. Authentication System — Deep Dive

Explain your authentication flow. Why JWT over sessions?

We use a dual-token JWT system:

  • Access Token: Short-lived (15 minutes), sent in every API request header (Authorization: Bearer), stored in sessionStorage on the frontend.
  • Refresh Token: Long-lived (30 days), sent only to the /auth/refresh endpoint, stored in an httpOnly cookie — completely inaccessible to JavaScript.

Why JWT over server-side sessions: Sessions require the server to store session state — in a distributed system (multiple server instances), every request must hit the same server or you need a shared session store (Redis). JWTs are stateless — any server can verify a token by checking the signature with the shared secret. This makes horizontal scaling trivial.

The JWT trade-off everyone asks about

JWTs can't be invalidated before expiry. If an access token is stolen, the attacker has 15 minutes of access — there's no way to "cancel" it. This is why we keep access tokens short-lived (15 min) and store them in sessionStorage (not localStorage — sessionStorage is cleared when the tab closes). The refresh token in httpOnly cookie is the real long-term credential — and we DO store a hash of it in Redis so we can invalidate it on logout.

Why store the refresh token in an httpOnly cookie instead of localStorage?

This is a critical security decision. localStorage is accessible to any JavaScript running on the page — including third-party scripts, browser extensions, or XSS injection. If an attacker injects malicious JavaScript, they can steal your refresh token with one line: localStorage.getItem('refresh_token').

An httpOnly cookie cannot be read by JavaScript at all — only the browser sends it automatically with requests to the matching domain. An XSS attack cannot steal it.

We also set Secure (only sent over HTTPS) and SameSite=Strict (not sent on cross-site requests, preventing CSRF) on the cookie.

The resulting security model

An XSS attacker can steal the access token from sessionStorage (15-min window). They cannot steal the refresh token. So their access expires in 15 minutes maximum. Compare this to localStorage with a 30-day refresh token — the attacker has 30 days of persistent access.

What is the email verification flow and why is it needed?

On registration, we generate a random secure token, store it hashed in the database, and "email" it to the user (in dev mode, we log it to the console). The user must submit this token to verify ownership of the email address.

Why: Without verification, anyone can register with someone else's email address. This has real consequences — the legitimate email owner starts receiving event notifications, booking confirmations, and marketing emails for an account they didn't create. Email verification proves ownership before any communication is sent.

We also block certain actions (like booking) until the email is verified — this reduces fake account creation since a verified email requires access to a real mailbox.

How does the forgot password / reset password flow work securely?

The flow: User submits email → server generates a random 32-byte hex token → stores the SHA-256 hash (not the raw token) in the database with a 1-hour expiry → sends the raw token to the user's email → user submits the token + new password → server hashes the submitted token and compares to the stored hash → if match, update password and delete the reset record.

Security details:

  • We store the HASH of the token, not the raw token — if the database is breached, the attacker can't use the stored hashes to reset passwords.
  • The response to POST /forgot-password is always the same message ("If this email exists, we've sent a reset link") regardless of whether the email is registered — this prevents email enumeration attacks where an attacker discovers which emails have accounts.
  • Tokens expire after 1 hour and are single-use (deleted on successful reset).

5. Real-time Architecture — Socket.io

How does your real-time messaging work? Why Socket.io?

We use Socket.io with the Redis adapter. Here's the architecture:

  1. When a user connects via WebSocket, they authenticate with their access token.
  2. The server verifies the JWT and joins the socket to a private room: user:<userId>.
  3. When any server-side event occurs (new message, booking confirmed, connection request), the server emits to the target user's room.
  4. The Redis adapter ensures all Socket.io events are broadcast across ALL server instances — if the sender is on Server 1 and the recipient is on Server 2, the message still arrives via Redis pub/sub.

Why Socket.io over raw WebSockets: Socket.io handles reconnection automatically, falls back to HTTP long-polling when WebSockets aren't available (corporate proxies, firewalls), provides rooms natively, and has excellent error handling. Raw WebSockets would require reimplementing all of this.

Trade-off

Socket.io is heavier than raw WebSockets — it adds ~100ms connection overhead for the handshake and transport negotiation. For a chat application, this is acceptable. For real-time gaming or financial trading, you'd use raw WebSockets or a protocol like QUIC.

Why do you need the Redis adapter for Socket.io?

Without the Redis adapter, Socket.io state is in-memory — only on the current server instance. If you run 3 server instances behind a load balancer:

  • User A's socket is on Server 1
  • User B's socket is on Server 2
  • When User A sends a message to User B, Server 1 tries to emit to User B's room — but User B isn't connected to Server 1, so the message is lost.

The Redis adapter solves this by using Redis pub/sub as a message bus. Server 1 publishes the event to Redis → Redis broadcasts to all subscribed servers → Server 2 receives it and delivers to User B's socket.

This is the standard pattern for horizontally scaling WebSocket servers. It's why we set up Redis from day one even before real-time features — we knew we'd need it.

6. Booking System — Race Conditions & Distributed Locks

This is the most interesting part — explain your seat booking system and how you handle concurrent bookings.

Seat booking is a classic distributed systems problem. Imagine 100 people trying to book the last 5 seats simultaneously. Without proper concurrency control, all 100 might read "5 seats available", all 100 might proceed to book, and you end up with 100 confirmed bookings for 5 seats — that's overselling.

Our solution: Redis distributed lock + Prisma transaction

Concurrent Request A: redis.SET seat-lock:eventId:sectionId "1" EX 10 NX → returns "OK" (lock acquired) → proceed to check seats in DB → Prisma $transaction: UPDATE event_sections SET available_seats = available_seats - N INSERT INTO bookings (...) INSERT INTO booking_participants (...) → redis.DEL seat-lock (release lock) Concurrent Request B (arrives 2ms later): redis.SET seat-lock:eventId:sectionId "1" EX 10 NX → returns NULL (lock already held) → immediately return 409 SEAT_LOCK_BUSY → client retries after a brief pause

Key details:

  • The SET NX EX command is atomic in Redis — "set this key only if it doesn't exist, and expire it automatically after 10 seconds". The atomicity is critical — you can't have a race condition in the lock acquisition itself.
  • The 10-second TTL is a safety net — if the server crashes holding the lock, it auto-releases after 10 seconds. Without this, a crashed server could hold the lock forever, blocking all future bookings.
  • We re-check seat availability INSIDE the lock — the check before the lock is an optimization (fail fast without acquiring the lock if seats are obviously gone), but the authoritative check is inside the transaction.
  • The Prisma $transaction ensures the seat decrement and booking creation are atomic — either both succeed or both roll back.
Trade-off we accepted

Our lock is at the section level (all requests for the same section queue behind the lock). At extreme scale (millions of concurrent bookings for a Taylor Swift concert), a single Redis lock becomes a bottleneck. The production solution would be a queue-based system where booking requests are queued in Redis lists (RPUSH/BLPOP) and processed sequentially — no lock contention, but higher latency.

What is a soft reservation and why did you implement it?

When a user creates a booking, their seats are immediately reserved (available_seats is decremented) and the booking enters PAYMENT_PENDING status with a 15-minute expiry (softReservationExpiresAt).

Why: Imagine going through a checkout flow — entering your payment details, your name, your address — only to find out at the final step that the seats were sold out because someone else confirmed payment 1 second before you. That's terrible UX. The soft reservation guarantees you have 15 minutes to complete payment before your reserved seats are released back to the pool.

What happens on expiry: A BullMQ job (booking-expiry queue) should fire when the reservation expires, reset the booking to CANCELLED, and increment available_seats back. We have the queue configured — the worker implementation would be wired in Phase 14.

Trade-off

Soft reservations temporarily reduce visible availability. If 50 people are in checkout flows, 50 seats appear "gone" to other users. This can make events look sold out when they're not. The mitigation is a reasonable short expiry (15 min) and a visible countdown timer in the UI. Ticketmaster uses this pattern — you see the "Your session expires in 10:00" countdown.

7. Caching Strategy — Redis

What do you cache and what do you not cache? How did you decide?

We cache data that is: read frequently, written infrequently, and where slightly stale data is acceptable.

Cache invalidation strategy

When an admin updates an event, we delete the event detail cache key. When a new event is created, we could invalidate all event list caches — but instead we let them expire naturally (2 min max staleness) since the new event will appear soon anyway. For seat counts specifically, we bypass cache entirely and always read from PostgreSQL.

Redis is doing three different jobs in your system. Isn't that a problem?

Redis serves as: (1) application cache, (2) Socket.io pub/sub adapter, (3) BullMQ job queue storage, (4) distributed lock storage, (5) refresh token store.

For development and early production, a single Redis instance is fine. The concern is that in production, if Redis goes down, all five functions fail simultaneously. The mitigation:

  • Redis Sentinel or Cluster for high availability — automatic failover if the primary goes down.
  • Separate Redis instances per concern at scale — one for cache (can lose data), one for queues (needs persistence), one for real-time (needs low latency). This is what companies like Stripe and Airbnb do.

For our scale (thousands of users, not millions), a single Redis with persistence (RDB snapshots) is the right call — simpler to operate, cheaper to run.

8. Background Jobs — BullMQ

Why use a job queue for ticket generation? Why not do it synchronously?

When a booking is confirmed, we need to: generate QR codes, create PDF tickets, send email confirmations to all participants, update booking status, potentially send push notifications. If we do all of this synchronously in the HTTP request, the API response could take 5–10 seconds. That's terrible UX.

Instead, the API response returns immediately ("Booking confirmed — tickets are being generated"), and a BullMQ job is queued to handle all the heavy lifting asynchronously. The user gets instant feedback; the tickets arrive a few seconds later.

Additional benefits of a queue:

  • Retry on failure: If the email service is temporarily down, the job retries with exponential backoff (2s, 4s, 8s...) automatically — no manual intervention.
  • Rate limiting: You can process at most N jobs per second, preventing overwhelming downstream services (email providers have API rate limits).
  • Observability: BullMQ keeps a record of completed, failed, and retrying jobs — you can see exactly what happened and why.
Trade-off

Eventual consistency — the user sees "confirmed" but doesn't have the ticket yet. This requires the UI to show a loading/generating state. For most users, this is fine. For high-anxiety scenarios (ticket gate opens in 5 minutes), you'd want synchronous generation with a tight timeout.

9. Frontend Architecture Decisions

Why TanStack Query (React Query) over Redux for server state?

Redux is a state management library — excellent for client state (UI state, user preferences, form state). But developers often use Redux for server state too (fetching, caching API responses), which creates significant boilerplate: actions, reducers, thunks/sagas, loading states, error states, cache invalidation logic — all manual.

TanStack Query is purpose-built for server state. It handles: fetching, background refetching, stale-while-revalidate, cache invalidation, loading/error/success states — all declaratively with useQuery.

Redux approach (server state): action: FETCH_EVENTS_REQUEST action: FETCH_EVENTS_SUCCESS (payload: events) action: FETCH_EVENTS_FAILURE (payload: error) reducer: handle all three cases thunk: async function with try/catch component: useSelector + useDispatch cache invalidation: manual React Query approach: const { data, isLoading, isError } = useQuery({ queryKey: ['events', filters], queryFn: () => api.get('/events', { params: filters }) }) // That's it. Cache, loading, error, refetch — all handled.
What we use Zustand for (client state)

Zustand manages the auth store: { user, accessToken, setUser, setToken, logout, hydrate }. This is true client state — it doesn't come from the server, it's derived from the login flow. Zustand is chosen over Redux because its API is much simpler (no actions, no reducers, just a store object with methods) and it's tiny (1KB vs Redux ~10KB + RTK).

Why Vite over Create React App?

Create React App (CRA) is deprecated. Vite is the modern standard. The practical difference:

  • Dev server start: CRA bundles everything before serving (~10-30 seconds). Vite serves files unbundled using native ES modules — the dev server starts in under 1 second.
  • Hot Module Replacement: CRA re-bundles on every save. Vite replaces only the changed module — HMR is near-instant.
  • Production build: Vite uses Rollup for production builds, which produces smaller bundles than CRA's Webpack config.

Why sessionStorage for the access token instead of localStorage or a cookie?

localStorage: Persists across browser sessions (tabs, closes, restarts). XSS attackers can steal it. Long-lived access tokens in localStorage are a known attack vector.

sessionStorage: Cleared when the tab closes. XSS can still steal it within a session, but the blast radius is smaller — the attacker's session ends with the tab. For a 15-minute access token, this is an acceptable trade-off. The refresh token (long-term credential) is in an httpOnly cookie — impossible to steal via XSS.

Memory only (no storage): The most secure option — the token lives in a JavaScript variable. Cleared on page refresh. Requires re-authentication every refresh. We chose sessionStorage as a middle ground — users don't have to log in on every page refresh, but tokens don't persist across browser restarts.

How do you handle URL-driven tab state — why not useState?

On the Connections and Admin pages, tabs are driven by URL query params (/connections?tab=requests) using React Router v6's useSearchParams hook — not useState. There are three concrete reasons why this matters:

  1. Deep-linking: The Notifications page navigates to /connections?tab=requests when a user clicks a connection request notification. With useState, the page always mounts on the default tab and ignores the query param. With useSearchParams, the tab is determined from the URL on every render.
  2. Refresh persistence: Refreshing the page keeps you on the same tab. With useState, a refresh always resets to tab 1.
  3. Browser history: Tab switches become navigable history entries — the back button works as expected.
useState (anti-pattern for tab state): - Notification navigates to /connections?tab=requests - Page mounts, useState initializes to 'connections' - Query param is ignored → user lands on wrong tab useSearchParams (correct pattern): - Notification navigates to /connections?tab=requests - Page mounts, reads searchParams.get('tab') → 'requests' - Correct tab selected immediately
Rule of Thumb

Use useSearchParams when: state must survive refresh, state can be set from outside the component (navigation), or state should be bookmarkable/shareable. Use useState for purely ephemeral UI state (modal open/close, hover state, form input).

How do you handle session expiry — what happens when a token expires mid-session?

We handle session expiry at the Axios interceptor level, so no individual page needs to handle 401s. The flow:

  1. Any API request that returns 401 Unauthorized triggers the response interceptor.
  2. The interceptor automatically calls POST /auth/refresh using the httpOnly cookie (the refresh token). This is attempted once per request (_retry flag prevents infinite loops).
  3. If the refresh succeeds: the new access token is stored in sessionStorage, the original request is retried with the new token — the user sees no interruption.
  4. If the refresh fails (e.g. refresh token also expired): show a toast.error('Session expired. Please sign in again.'), then redirect to /login after 1500ms. The delay ensures the user can read the message before being redirected.
API request → 401 → try POST /auth/refresh (httpOnly cookie) → success: store new token → retry original request → user sees nothing → fail: toast.error('Session expired') → setTimeout 1500ms → /login
Why toast + delay instead of immediate redirect?

Immediately redirecting on 401 is jarring — the user's current work disappears without explanation. The 1500ms delay with a visible toast gives enough time to read the message while being fast enough to not feel broken. This is the UX pattern used by most production apps (GitHub, Notion, etc.).

How do you keep data fresh across pages — explain React Query cache invalidation strategy.

React Query caches API responses by query key. The challenge: when action on Page A affects data shown on Page B, Page B needs to refetch. We handle this with queryClient.invalidateQueries() called in mutation onSuccess callbacks.

Concrete example — sending a connection request from EventDetail:

User clicks "Connect" on EventDetail /events/123 → POST /connections/events/123/users/456 (mutation) → onSuccess: toast.success('Connection request sent!') queryClient.invalidateQueries(['connections-sent']) → User navigates to /connections?tab=sent → useQuery(['connections-sent']) sees its cache is stale → refetches automatically → shows the new request immediately

Notification badge updates: The sidebar badge for notifications is driven by useQuery(['notif-unread-count']). When a new notification arrives via Socket.io, the socket.on('notification:new') handler calls queryClient.invalidateQueries(['notif-unread-count']) — the badge updates in real time without polling. A 60-second interval fallback covers missed socket events.

invalidateQueries vs setQueryData

invalidateQueries marks the cache as stale and triggers a background refetch on next render. setQueryData directly mutates the cache with optimistic data. We use invalidateQueries because the backend is the source of truth — we refetch rather than guess. Optimistic updates would be appropriate for toggle actions (like/unlike) where the new state is known before the API responds.

How did you handle isAdmin access control — from database to UI?

Admin access is enforced at three layers — each is necessary, none is sufficient alone:

LayerWhat it doesWhat happens if missing
Databaseis_admin: boolean column on users tableNo persistent record of who is admin
Backend middlewarerequireAdmin middleware checks req.user.isAdminAny authenticated user can call admin endpoints
Frontend sidebar{user?.isAdmin && <AdminLink />}Admin link hidden but endpoints still accessible via direct URL
Frontend routeProtectedRoute wraps admin page — non-admins get 403 from API anywayAdmin page renders but all queries return 403

The key bug we encountered: isAdmin was not included in the Prisma select in the login service. The field was in the DB and in the Zod schema, but never fetched — so the Zustand auth store always had isAdmin: undefined, which is falsy. The sidebar condition never rendered the link. The fix was adding isAdmin: true to the login select and isAdmin: boolean to the AuthUser interface.

Important: Session Data is Stale

The Zustand auth store holds the user object from login time. If isAdmin is toggled in the DB after a user is already logged in, they won't see the change until they sign out and sign in again. For production: add a GET /auth/me call on app load that refreshes the user object in the store.

What is the bcrypt cost factor and how did you choose 12?

bcrypt's cost factor (also called work factor) determines how computationally expensive hashing is. Cost factor 12 means the algorithm performs 2^12 = 4,096 iterations. Each increment doubles the time — cost 13 takes twice as long as cost 12.

At cost 12, hashing takes approximately 250–400ms on a modern CPU. This means an attacker with a stolen database can only attempt ~2-4 password guesses per second per CPU core — making brute force infeasible for strong passwords.

The trade-off: every login costs 250ms in CPU time. At 1,000 concurrent logins/second, that's significant CPU load. Companies like GitHub use cost 10-11 for this reason. Cost 12 is the OWASP recommendation for 2024 and appropriate for most applications.

11. Scalability — What Could Break & How to Fix It

What are the bottlenecks in your current architecture?

Being honest about limitations is what separates senior engineers from junior ones. Here are the real bottlenecks:

1. Single Redis instance

Redis handles cache, queues, locks, pub/sub, and token storage. A single Redis failure takes down all five functions simultaneously. Fix: Redis Sentinel (primary-replica with automatic failover) or Redis Cluster.

2. Seat lock serialization

Our distributed lock processes one booking per section at a time. For a Coldplay concert with 50,000 concurrent booking attempts, this is a severe bottleneck. Fix: Move to a database-level optimistic concurrency control with SELECT FOR UPDATE SKIP LOCKED — PostgreSQL can handle this with much higher throughput than a Redis lock.

3. No database read replicas

All reads and writes go to the same PostgreSQL instance. As read traffic grows (event browsing, user profiles), this becomes a bottleneck. Fix: PostgreSQL streaming replication — writes go to primary, reads go to replicas. Prisma supports read replicas via the @prisma/extension-read-replicas extension.

4. N+1 query risk

Prisma's include prevents N+1 queries in most cases, but complex nested includes can generate multiple queries. For the event list (12 events × sections × venue), this is currently 1 well-structured query. At scale, this would benefit from query analysis and potentially denormalization of frequently-read fields.

5. No CDN for static assets

Event banner images are served via Cloudinary URLs (good — Cloudinary has CDN built in), but the frontend static assets (JS, CSS) are served directly from the Node.js server in development. In production, the frontend should be deployed to a CDN (Vercel, Cloudflare Pages, AWS CloudFront).

How would you scale this to 1 million users?

This is a common system design question. Here's the honest answer for EventMeet specifically:

  1. Database: PostgreSQL → AWS RDS with read replicas + connection pooling via PgBouncer. For the social graph (connections, availability), consider migrating to a graph database (Neo4j) where traversal queries are O(1) instead of O(n joins).
  2. Cache: Redis Cluster (horizontal sharding). CDN layer (CloudFront) for event data — cache at the edge, 100ms response globally.
  3. API: Multiple Node.js instances behind an Application Load Balancer. Socket.io still works via Redis adapter.
  4. Queues: BullMQ workers on separate machines — scale ticket generation independently from the API.
  5. Search: PostgreSQL ILIKE is too slow at scale. Migrate event search to Elasticsearch or Typesense — full-text search with filters in <10ms.
  6. Payments: Add idempotency keys to all payment operations — prevent duplicate charges on retry.
  7. Notifications: At scale, switch from Socket.io push to a service like Firebase Cloud Messaging (FCM) for mobile push notifications.

12. Bugs Encountered & What We Learned

This section is gold in interviews. Most candidates only talk about what worked. Talking about bugs you found and fixed shows debugging ability, self-awareness, and real project experience.

What bugs did you encounter during development?

Bug 1: Prisma Schema Validation Error (Phase 1)

What happened: The initial schema had circular relation references that Prisma couldn't resolve — the User model referenced ConnectionRequest as both sender and receiver, and Prisma needed explicit @relation names to disambiguate.

Fix: Added named relations: @relation("connectionSender") and @relation("connectionReceiver"). The lesson: when a model has multiple relations to the same model, Prisma requires explicit names on all of them.

Bug 2: ParticipantStatus Enum Mismatch (Phase 7)

What happened: When creating a booking with participants, we used status value 'CONFIRMED' for the initiator — but the ParticipantStatus enum in Prisma didn't have CONFIRMED. It had ACCEPTED. Runtime error on first booking attempt.

Fix: Changed to 'ACCEPTED' for the initiator, 'INVITED' for participants.

Lesson: Always cross-reference enum values between the Prisma schema and application code. TypeScript would have caught this if we had used the Prisma-generated enum type instead of a string literal.

Bug 3: Events Response Shape Mismatch (Phase 13 — Frontend)

What happened: The frontend expected res.data.data.events (nested events array) but the backend's sendPaginated() function returns the array directly as res.data.data. The page was blank with Cannot read properties of undefined (reading 'filter').

Fix: Changed frontend query to: return { data: res.data.data, pagination: res.data.pagination }

Lesson: Always verify API response shapes at integration time. In a shared type system (Zod + TypeScript from backend to frontend), this class of error is entirely preventable.

Bug 4: user.fullName null crash (Phase 13 — Frontend)

What happened: The available-users endpoint returned users where fullName could be null (profile not yet filled). The frontend did user.fullName.charAt(0) — crash.

Fix: Changed to user.fullName?.charAt(0)?.toUpperCase() ?? '?' — optional chaining with a fallback.

Lesson: Always use optional chaining when accessing properties from external data sources. Data from APIs can always have unexpected null/undefined values regardless of what your TypeScript types say.

Bug 5: Docker Desktop WSL Timeout (Environment)

What happened: Docker Desktop on Windows failed to start with "WSL distro installation timeout". The WSL process itself was hanging.

Fix: wsl --shutdown + wsl --update + restart Docker Desktop as Administrator.

Lesson: Docker on Windows has more friction than on Linux/Mac due to the WSL2 layer. Understanding the WSL2 architecture (a full Linux VM running inside Windows Hyper-V) explains why these issues occur and how to diagnose them.

Bug 6: isAdmin Not Returned from Login — Admin Dashboard Invisible

What happened: After building the Admin Dashboard and setting is_admin = true in the DB, the Admin link never appeared in the sidebar — even after re-login. Three-layer omission: isAdmin not in Prisma select → not in AuthUser type → Zustand store held isAdmin: undefined → sidebar condition always false.

Fix: Added isAdmin: true to the login service Prisma select and isAdmin: boolean to the AuthUser interface. Required sign-out + sign-in to pick up the change (Zustand store is populated at login time).

Lesson: Auth fields not in the Prisma select are silently undefined. Always trace the full chain when a field is missing: DB → select → type → store → component.

Bug 7: URL-Driven Tab State — Notifications Deep-Link Broke

What happened: Clicking a connection request notification navigated to /connections?tab=requests but the user always landed on the "Connections" tab, not "Requests". The tab was stored in useState which ignores the URL entirely.

Fix: Replaced useState with useSearchParams on both Connections and Admin pages. Tab is now read directly from the URL: searchParams.get('tab') || 'connections'.

Lesson: Any state that can be set by something outside the component (navigation, deep-links, external triggers) must live in the URL, not in component state.

Bug 8: sendPaginated extra Field — Notifications Blank Despite DB Data

What happened: Notifications page showed blank even with 2 records in DB. Frontend read res.data.notifications (doesn't exist) and res.data.unreadCount (also doesn't exist). The sendPaginated response always puts data at res.data.data and optional fields at res.data.pagination.extra.

Fix: Changed to res.data.data for the array and res.data.pagination?.extra?.unreadCount for the count.

Lesson: sendPaginated with extra data buries it inside pagination.extra, not at the response root. This is the fifth time this shape was misread across the project — a shared type or documented contract prevents it.

13. System Design — If Asked to Design This From Scratch

Design a seat booking system that handles 10,000 concurrent users trying to book the last 5 seats.

This is a direct application of what we built. Here's the answer structured for a system design interview:

Step 1: Identify the core problem

Seat booking is a race condition problem. The constraint: available seats can never go below zero. The challenge: 10,000 concurrent reads see "5 seats available" and all try to write.

Step 2: Options and trade-offs

ApproachHow it worksProsCons
Optimistic LockingRead seats + version number. Update only if version matches. Retry on mismatch.High throughput — no blockingMany retries under heavy contention — high DB load
Pessimistic Locking (SELECT FOR UPDATE)Lock the row in the DB on readGuaranteed no oversellLocks held across entire transaction — slow under load
Redis Distributed Lock (our approach)Redis SET NX EX before DB writeFast, atomic, auto-expirySingle lock = serialization bottleneck at extreme scale
Queue-based allocationAll booking requests join a Redis queue. Worker processes one at a time.Perfect fairness, no contentionHigher latency — users wait in queue

Step 3: For 10,000 concurrent users — Queue-based

At extreme scale, switch from distributed lock to a queue: RPUSH booking-queue:sectionId {userId, timestamp}. A worker pops requests with BLPOP and processes them sequentially. Users see "position X in queue". This is literally how Ticketmaster works for high-demand events.

Design the notification system for this application.

Our notification system has two channels:

  1. Real-time (Socket.io): When the server creates a notification record in PostgreSQL, it immediately emits a notification:new event via Socket.io to the user's room. If the user is online, they see it instantly.
  2. Persistent (database): Notifications are stored in the notifications table. When a user opens the app after being offline, they fetch missed notifications via GET /notifications.

This is the correct hybrid approach: real-time when the user is online, pull-based catch-up when they return. The 13 notification types cover every significant event in the user journey (connection requests, booking updates, group invites, rating reminders).

What we'd add for production

Mobile push notifications (FCM/APNs) for when the app is closed. Email digests for users inactive for 24+ hours. Notification preferences (let users disable certain types). Currently the architecture is ready for these — the notification creation is centralized in notifications.service.ts, so adding a push notification call is a 5-line change.

Why did you choose this particular set of technologies rather than a simpler stack?

This is a great question to answer honestly. Every technology in this stack was chosen for a specific reason tied to a specific requirement:

  • PostgreSQL: Because we have relational data with strict integrity requirements (bookings, payments, seats). MongoDB would have worked for social features but failed on financial data.
  • Redis: Because we needed caching (performance), distributed locks (seat booking correctness), pub/sub (Socket.io at scale), and job queues (BullMQ). All Redis use cases emerged from real requirements.
  • Socket.io: Because real-time messaging is a core feature, not an afterthought. Building it into the architecture from day one is far easier than retrofitting.
  • BullMQ: Because email delivery, ticket generation, and refund processing should not block API responses and need retry capabilities.
  • Zod: Because validation should be the same on both sides of the API — the same schema can be used in the Express middleware AND in React Hook Form, preventing API contract drift.

The key insight: we didn't choose technologies because they're trendy. We identified the requirements first (real-time, concurrency control, async jobs, type safety) and then chose the tools that best solved those specific requirements.
